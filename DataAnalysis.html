First steps in Linux shell

In linux, all the operations can be performed through a command-line terminal (also known as shell), using a scripting language called Bash.

The overall structure is simple:

command input

So, every operations is called by a command that can receive an input , written after a space.

not all commands have mandatory inputs. for example, ls command can run with no inputs, and geves, as output, the contents of the current directory.

ls

Commands can also have options, which must be defined before the input and following a single or a double - (options 1 character long with a single - and for longer options -- ).

Some example structures:

command -o input
command --option input
command -o --option input

Options can be used to define behaviour of the command. for example, in ls -a , the -a option means show also hidden files.

Options can also be input with a value. for example, --hide options in ls says do not show contents with a certain pattern. value of this option is a pattern which must be written after the option. as an example:

ls --hide "*.zip"

From now on, when a value must be specified and is not a fixed part of the code, its value will be specified as something like <NAME-OF-VALUE . So general usage for the previous code could be shown as 

ls --hide <PATTERN>

And the general structure for most commands can be seen as:

command -o --options <VALUE> <MAIN-INPUT>

Finding patterns in files

By using the grep command it is possible to find the occurrences of pattern in a given file. General use is:

grep pattern filename

e.g. to find occurrences of CAGATA in minigenome.fa, the command is:

grep CAGATA minigenome.fa

or

grep "CAGATA" minigenome.fa

some useful -options:

the -c option counts the occurrences. so to count how many CAGATAs are there in the first part of human genome, the command is:

grep -c "CAGATA" minigenome.fa

the -n option prints, before the occurrences, the number of line containing them. to show all the lines containing CAGATA in minigenome.fa, use:

grep -n "CAGATA" minigenome.fa

So lets use pattern matching for eploring the structure of the genome fasta file. in a FASTA file, each DNA sequence is preceded by a line containing > character and the name of the DNA sequence.

Saving values as variables

a variable is a position in the memory, identified by a name, and containing data, usually single values. to save a variable, the syntax is:

variableName=value

note there is no space between the name of the variable and the =. otherwise, the name of the variable would have been interpreted as a command!

as a first example, generate a variable called bob containing the value 5 The code for it is:

bob=5

in linux, we cannot just write the name of a variable to print it (as it works in R, for example), because almost all code is interpreted as command-space-input. to print something, we can use the echo command. to print bob the code is:

echo bob

this prints bob characters, but not the value in the variable bob. To refer to the stored value, we need to use $variableName, so:

echo $bob

Use of $ in Linux shell

In linux shell, $ stands for make some operation and write the result in the code.

1. First use: alone before text $text

the meaning is take the value stored in variable called text and substitute it in the code.

example:

file=minigenome.fa
nLines=4

writing

head -n$nLines $file

is equivalent to write:

head -n4 minigenome.fa

2. Second use: after single brackets $(code)

the meaning is evaluate the code inside the brackets and substitute it in the rest of the code.

Most common use of this is to create variables as results of code.

example: writing

var=$(grep -c "CACCA" minigenome.fa)

given that grep -c "CACCA" minigenome.fa generates 18044, this code is equivalent to write:

var=18044

3. Third use: after double nested brackets $((arithmetic operation))

the meaning is calculate the operation inside the brackets and substitute it in the rest of the code.

Most common use of this is to create variables as results of code.

example:

tail -n$((2+2)) minigenome.fa

is equivalent to write:

tail -n4 minigenome.fa

Gtf file filtering

Gtf files are tsv (tab-delimited-values) tables containing annotations of a genome. annotations are arranged as rows containing info about the chromosome in which they are, their start and end position, the strand and type and name of annotation (es a gene, an exon, a promoter)

the structure for each row is:

seqname source  feature start   end     score   strand  frame   attributes
chr1    HAVANA  gene    1001138 1014540 .       +       .       gene_id "ENSG00000187608.10"; gene_type "protein_coding"; gene_name "ISG15"; level 2; hgnc_id "HGNC:4053";

It is possible to filter the annotations using grep. For example, we can get all annotations regarding MT-ND6 gene by the code

grep "MT-ND6" annotation.gtf

Chaining commands: pipelines

Filtering files is a common situation where we may want to perform multiple sequential operations. For example, we might want to filter all genes belonging to a certain chromosome, then all the protein_coding, and then visualizing the first n occurrences.

we can perform this by generating intermediate files:

# Step 1: Filter genes from a specific chromosome (e.g., chrX)
grep "chrX" annotation.gtf > step1.gtf

# Step 2: Filter for 'protein_coding' entries from the previous file
grep "protein_coding" step1.gtf > step2.gtf

# Step 3: Display the first n lines of the result
head -n 10 step2.gtf

In Linux shell, pipes (|) are used to combine commands so the output of one command serves as the input for the next. This allows for efficient data processing without creating intermediate files. Chains of commands linked by pipes are called pipelines.

It is similar to > , which sends the output of a command to a file. In this case it sends the output as the input of another command. general usage:

command1 input | command2

the example up here, using pipes, would be:

grep "chrX" annotation.gtf | grep "protein_coding" | head -n 10

Identify gRNA flanking regions and extract variable part

Removal of flanking regions can be done using cutadapt defining the right options.

cutadapt -g <PATTERN> --maximum-length <LENGTH> --times <NUM> --rc -o <OUTPUT_FILE> <INPUT_FILE>
cutadapt -g <PATTERN> \
--maximum-length <LENGTH> \
--times <NUM> \
--rc \
-o <OUTPUT_FILE> \
<INPUT_FILE>

Meaning of the options:

  • -g : The sequence to trim from the 5' end of the reads (e.g., an adapter or primer).
  • --maximum-length : Maximum allowed length of reads after trimming.
  • -- times : Number of times to search for and trim the adapter sequence in each read.
  • -rc : Search also in the reverse complement of the sequence.
  • -o : Name of the trimmed output FASTQ file.
  • <INPUT_FILE>: name of the input FASTQ file to process.

Alignment of the reads to human genome

To align the reads, Bowtie2 needs to remodel (or build) the genome in a more efficient format using BurrowsWheeler transform. the command bowtie2-build receives a genome FASTA file as input and generates a series of files containing the remodelled version of the genome. each output file share a prefix in the name, which is defined in the code. Common usage is:

bowtie2-build <FASTA-FILE> <OUTPUT-PREFIX>

Once the genome is built, it can be used to map reads from a sequencing FASTQ file using bowtie2 command. Common usage is:

bowtie2 -x <GENOME-PREFIX> -U <FASTQ-FILE> -S <OUTPUT-SAM-FILE>

Bowtie2 maps all the sequences of the input FASTQ file to the genome, returning a SAM file containing all the information about if and where each sequence mapped in the genome.

Output of Genome Alignment: SAM Format

The SAM (Sequence Alignment/Map) format is the standard for representing the alignment of sequencing reads to a reference genome. It is a tab-separated values (TSV) format (just like gtf files), where each row corresponds to a read alignment.

Here an example for the structure of a SAM file:

QNAME  FLAG  RNAME  POS   MAPQ  CIGAR RNEXT PNEXT TLEN  SEQ       QUAL       OPT
read1  16    chr18  100   60    8M    *     0     0     AGCTTAGC  1;;=@>67=  AS:i:0
...

Analyzing the Alignment Results

Count Reads Aligned to a Specific Chromosome (in my example chr18)

cat aligned.bam | grep -c "chr18"

Keeping chromosome and read sequence only and visualising first lines

cut -f3,10 aligned.bam | head

Using them to count occurrences of mapped sequences

cut -f3,10 aligned.bam | grep chr | sort | uniq -c

A final sort lead to the most frequent aligned sequences

cut -f3,10 aligned.bam | grep chr | sort | uniq -c | sort -n | tail -n20

Converting SAM to BAM for efficiency

BAM files are equivalent to SAM, but data is binary coded instead to be text. This reduces storage use and allows other programs for counting, annotating or visualizing alignments to be more efficient.

samtools can perform various operation of SAM/BAM files.

To convert SAM to BAM:

samtools view -b -o <OUTPUT-BAM-FILE> <INPUT-SAM-FILE>

other operations needed by downstream analysis tools are sorting reads and generating an index file (bam.bai)

to sort BAM file:

samtools sort -o <OUTPUT-BAM-FILE>  <INPUT-BAM-FILE>

to generate ad index:

samtools index <INPUT-BAM-FILE>

A sorted BAM file and a BAM.BAI file are enough, for example, to visualize graphically the aligment in softwares like IGV

https://igv.org/app/

Annotating and Counting Alignments

When only a few reads map to the genome, as in this case, identifying alignments can feel like searching for a needle in a haystack. Additionally, it is often crucial to systematically annotate the positions of aligned reads and count how many fall within each gene or similar features.

One of the standard tools for this task is featureCounts from the Subread package. It requires a BAM file as input and a GTF file containing the annotations of the genome used for alignment.

The -O option allows reads to be assigned to overlapping features, and the -g option specifies which attribute from the GTF file is included in the final output. By default, this attribute is gene_id, but it can be customized to use others, such as gene_name.

General Usage

featureCounts -a <ANNOTATION-GTF> \
              -g <ATTRIBUTE> \
              -O \
              -o <OUTPUT-FILE> \
              <INPUT-BAM-FILE>

Output data

The output of featureCounts is a TSV (tab-separated values) table containing, for each gene, the chromosome it resides on, its coordinates, and the number of occurrences (read counts). This output represents the conclusion of the primary data analysis, where raw data is transformed into a structured matrix or table that can be further imported in R or Python for subsequent analyses, also referred as secondary data analysis.

Anyway, it is possible to perform simple operations using basic Linux commands. For example, it is possible to count the number of genes which at least a read of gRNA mapped onto it.
